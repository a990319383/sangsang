

# 数据变更字段自动填充

### 零侵入式字段默认值自动维护解决方案

---

## 一、使用场景

> 新增、修改时自动维护固定字段值
>
> 常见于新增修改时间，操作人。租户id，组织权限等字段的默认维护

### 核心价值

```
┌─────────────────────────────────────────────────────┐
│                                                          │
│  ✅  无需手动设置字段值                                │
│  ✅  数据变更时自动维护                                    │
│     • INSERT 时自动填充                                     │
│     • UPDATE 时自动更新                                     │
│     • 无需修改业务代码                                     │
│                                                          │
└─────────────────────────────────────────────────────┘
```

## 三、工作原理

> 框架基于 MyBatis 拦截器 + JSQLParser 解析 SQL，自动在 INSERT/UPDATE 时维护字段默认值

```
┌─────────────────────────────────────────────────────┐
│                                                          │
│  业务代码执行 INSERT/UPDATE                               │
│         ↓                                                │
│  MyBatis 拦截器拦截                                   │
│         ↓                                                │
│  识别表中需要维护默认值的字段                             │
│         ↓                                                │
│  执行策略获取默认值                                       │
│         ↓                                                │
│  自动改写 SQL，添加字段和值                                │
│         ↓                                                │
│  输出改写后的 SQL 执行                                   │
│                                                          │
└─────────────────────────────────────────────────────┘
```

### 效果示例

> 场景：订单表需要在新增时自动记录创建部门

**原始 SQL**
```sql
INSERT INTO 订单表(订单编号, 金额)
VALUES ('ORD001', 1000.00)
```

**自动填充后**
```sql
INSERT INTO 订单表(部门字段, 订单编号, 金额)
VALUES ('销售部', 'ORD001', 1000.00)
```

## 五、快速接入

### Step 1: 引入依赖

```xml
<!-- 核心依赖 -->
<dependency>
    <groupId>io.gitee.tired-of-the-water</groupId>
    <artifactId>sangsang-core</artifactId>
    <version>${version}</version>
</dependency>
```

### Step 2: 开启配置

> 💡 **提示**
>
> • 框架会从 scanEntityPackage 扫描的路径中缓存表字段信息
>
> • 支持配合数据权限隔离功能，实现租户等场景

```properties
# 配置 @TableName 标注的实体类扫描路径
sangsang.scanEntityPackage[0]=com.sangsang.*.entity

# 开启数据变更时设置默认值
sangsang.fieldDefault.enable=true
```

### Step 3: 自定义默认值策略

```java
package xxx.strategy;

import com.sangsang.domain.enums.SqlCommandEnum;
import com.sangsang.domain.strategy.fieldDefault.FieldDefaultStrategy;
import org.springframework.stereotype.Component;

/**
 * 创建时间默认值策略
 */
@Component
public class CreateTimeStrategy implements FieldDefaultStrategy<LocalDateTime> {

    @Override
    public boolean whetherToHandle(SqlCommandEnum sqlCommandEnum) {
        // 新增时设置默认值
        return SqlCommandEnum.INSERT.equals(sqlCommandEnum);
    }

    @Override
    public LocalDateTime getDefaultValue() {
        return LocalDateTime.now();
    }
}
```

### Step 4: 字段标注

> 标注在实体类字段上，支持在基类中统一标注

```java
/**
 * 实体基类
 */
public class BaseEntity {

    @TableId(type = IdType.AUTO)
    private Long id;

    /**
     * 创建时间 - 新增时自动维护
     */
    @FieldDefault(CreateTimeStrategy.class)
    private LocalDateTime createTime;

    /**
     * 创建人 - 新增时自动维护
     */
    @FieldDefault(CreateByStrategy.class)
    private String createBy;

    /**
     * 修改时间 - 新增和修改时自动维护
     */
    @FieldDefault(UpdateTimeStrategy.class)
    private LocalDateTime updateTime;

    /**
     * 修改人 - 新增和修改时自动维护
     */
    @FieldDefault(UpdateByStrategy.class)
    private String updateBy;
}
```

### Step 5: 验证

```bash
# 所有涉及订单表的新增/修改 SQL
# 自动维护 create_time、update_time、create_by、update_by ✅
```

### 强制覆盖原 SQL 的值

> 当需要设置的值在原业务 SQL 中存在，强制使用策略值覆盖

```properties
# 进行字段标注时，使用 mandatoryOverride 为 true
```

```java
/**
 * 强制覆盖原 SQL 值
 */
@FieldDefault(
    value = CreateTimeStrategy.class,
    mandatoryOverride = true  // 强制使用策略值，忽略 SQL 中的值
)
private LocalDateTime createTime;
```

---

### 配合数据权限隔离实现租户自动写入

> 配合数据权限隔离功能，可实现租户数据的自动管理和查询隔离

```java
/**
 * 租户 ID 默认值策略
 */
@Component
public class TenantIdStrategy implements FieldDefaultStrategy<Long> {

    @Override
    public Long getDefaultValue() {
        // 从当前登录用户的 ThreadLocal 中获取租户 ID
        return UserContext.getTenantId();
    }
}
```

```java
@TableName("tb_order")//这个类没必要用于mapper，service的入参/返回值
@Data
// 使用数据权限隔离
@DataIsolation
public class OrderEntity {
	// 自动维护租户 ID
	@FieldDefault(TenantIdStrategy.class)
	private Long tenantId;
}
```

## 七、不兼容场景

### 通用限制

| 限制项 | 说明 |
|:---|:---|
| JSQLParser 语法 | 基于 jsqlparser4.9 解析，不支持的语法无法兼容 |
| MyBatis-Plus saveBatch() | service 层自带方法不支持 |
| INSERT VALUES 全字段 | 表名后未跟具体字段（开启 autoFill 后可兼容） |

### 特殊语法场景

| 场景 | 说明 |
|:---|:---|
| INSERT...SELECT | 子查询返回值无法自动填充 |

## 八、常见问题

### Q1: 如何区分 INSERT 和 UPDATE 的默认值？

```java
@Override
public boolean whetherToHandle(SqlCommandEnum sqlCommandEnum) {
    // sqlCommandEnum 包含 INSERT, UPDATE, DELETE, SELECT
    if (sqlCommandEnum == SqlCommandEnum.INSERT) {
        return true;  // INSERT 时维护
    }
    if (sqlCommandEnum == SqlCommandEnum.UPDATE) {
        return true;  // UPDATE 时维护
    }
    return false;  // 其他情况不维护
}
```

---

### Q2: 如何实现"创建人等于修改人"？

```java
/**
 * 修改人默认值策略
 * 新增和修改时都设置为当前登录用户
 */
@Component
public class OperatorStrategy implements FieldDefaultStrategy<String> {
    @Override
    public boolean whetherToHandle(SqlCommandEnum sqlCommandEnum) {
        // sqlCommandEnum 包含 INSERT, UPDATE, DELETE, SELECT
        if (sqlCommandEnum == SqlCommandEnum.INSERT) {
            return true;  // INSERT 时维护
        }
        if (sqlCommandEnum == SqlCommandEnum.UPDATE) {
            return true;  // UPDATE 时维护
        }
        return false;  // 其他情况不维护
    }
    
    @Override
    public String getDefaultValue() {
        return UserContext.getCurrentUserId();
    }
}
```
