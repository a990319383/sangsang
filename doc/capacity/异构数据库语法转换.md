

# 异构数据库语法转换

>  零侵入式异构数据库迁移解决方案

---

## 一、使用场景

> 信创大环境下，项目从 MySQL 迁移到达梦等信创数据库
>
> 老Oracle项目因为费用问题想切换到mysql数据库
>
> mysql项目因为性能问题想切换到其它数据库
>
> 项目已经存在，但是迁移到异构数据库后，原项目的sql语法很多不兼容，此时就可以考虑用此功能了

### 核心价值

```
┌─────────────────────────────────────────────────────────────┐
│                                                          │
│  ✅  无需手动修改业务 SQL                                 │
│  ✅  sql执行时语法自动转换                                │
│     • MySQL 自动转换为达梦语法                             │
│     • 支持扩展其他数据库转换                               │
│     • 一处配置，全项目生效                                 │
│                                                          │
└─────────────────────────────────────────────────────────────┘
```

## 二、方案对比

| 对比项 | 硬改项目 SQL | **field-encryptor** |
|:---:|:---:|:---:|
| **改造方式** | 满项目查找修改 | **引入依赖 + 配置** |
| **工作量** | 巨大 | **极小** |
| **覆盖率** | 难以确保全部改完 | **全局自动转换** |
| **维护成本** | 新增语法需手动处理 | **框架统一处理** |
| **风险** | 容易遗漏出错 | **低风险** |

## 三、工作原理

> 框架基于 JSQLParser 对 SQL 进行语法解析，自动将源数据库语法转换为目标数据库语法

```
┌─────────────────────────────────────────────────────────────┐
│                                                          │
│  业务代码执行 MySQL 语法的 SQL                              │
│         ↓                                                │
│  MyBatis 拦截器拦截                                   │
│         ↓                                                │
│  JSQLParser 解析 SQL 语法树                             │
│         ↓                                                │
│  遍历语法树，转换为目标数据库语法                         │
│         ↓                                                │
│  输出转换后的 SQL 执行                                   │
│                                                          │
└─────────────────────────────────────────────────────────────┘
```

### 效果示例

> 原始 SQL（MySQL 语法）

```sql
-- 查询语句
SELECT `user_name`, `phone` FROM `tb_user` WHERE `phone` = ?

-- 日期函数
SELECT DATE_FORMAT(create_time, '%Y-%m-%d') FROM tb_user

-- LIMIT 分页
SELECT * FROM tb_user LIMIT 10, 5

-- 反引号标识符
SELECT * FROM `tb_user` WHERE `user_name` LIKE 'admin%'
```

> sql执行时，自动转换为达梦语法

```sql
-- 查询语句
SELECT "user_name", "phone" FROM "tb_user" WHERE "phone" = ?

-- 日期函数
SELECT TO_CHAR(create_time, 'YYYY-MM-DD') FROM tb_user

-- LIMIT 分页
SELECT * FROM tb_user LIMIT 5 OFFSET 10

-- 双引号标识符
SELECT * FROM "tb_user" WHERE "user_name" LIKE 'admin%'
```

## 五、快速接入

### Step 1: 引入依赖

```xml
<!-- 核心依赖 -->
<dependency>
    <groupId>io.gitee.tired-of-the-water</groupId>
    <artifactId>encryptor-core</artifactId>
    <version>${version}</version>
</dependency>
```

### Step 2: 开启配置

> 💡 **提示**
>
> • 完整切换数据库还需修改驱动和连接地址

```properties
# 选择语法转换模式：mysql2dm，这里是mysql转达梦，oracle2mysql仍在开发中
field.transformation.patternType=mysql2dm
```

### Step 3: 切换数据库

> 修改项目的数据库驱动和连接地址

```properties
# MySQL 配置（原有）
#spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver
#spring.datasource.url=jdbc:mysql://localhost:3306/db

# 达梦配置（切换后）
spring.datasource.driver-class-name=dm.jdbc.driver.DmDriver
spring.datasource.url=jdbc:dm://localhost:5236/db
```

### Step 4: 验证

```bash
# 无需修改任何业务 SQL
# 所有 MySQL 语法的 SQL 自动转换为达梦语法 ✅
```

### 自定义语法转换器

> 当框架遗漏了某些语法的转换，想在项目中自行扩展

#### 实现步骤

```
步骤1: 创建转换器包路径
  包路径格式：com.sangsang.transformation.{patternType}
  示例：com.sangsang.transformation.mysql2dm

  注意：patternType 对应配置的 field.transformation.patternType

步骤2: 实现对应基类转换器
  • 列转换：implements ColumnTransform
  • 表转换：implements TableTransform
  • 函数转换：implements FunctionTransform
  • 表达式转换：implements ExpressionTransform
  • 其它各种基类转换器详见com.sangsang.transformation 包下implements TransformationInterface的各种抽象类

  参考源码：com.sangsang.transformation.mysql2dm 包下实现
```

#### 示例代码

```java
package com.sangsang.transformation.mysql2dm;

import com.sangsang.transformation.core.ColumnTransform;
import com.shade.net.sf.jsqlparser.schema.Column;
import org.springframework.stereotype.Component;

/**
 * 自定义列转换器示例
 */
@Component
public class CustomColumnTransform implements ColumnTransform {

    @Override
    public Column transform(Column column) {
        // TODO: 实现自定义列转换逻辑，这里最好在原有的入参对象上面改，返回原入参对象
        return column;
    }
}
```

#### 特殊场景

> 某些语法转换时，会涉及到使用到sql的入参或者移除掉sql的入参（比如分页的语法转换）

```
- 在转换器中获取到的sql，所有的 ? 占位符会被替换为 
	com.sangsang.domain.constants.FieldConstant#PLACEHOLDER + 自增序列
- 通过这种方式就可以精确获取到第几个 ？ 占位符的值
```

- 获取当前sql入参的某个?占位符的入参值

> 当语法转换逻辑中需要获取到?的具体的值时，可以通过下面方法获取

```
com.sangsang.domain.context.TfParameterMappingHolder#getParameterMapping
```

- 转换后的sql需要移除掉?占位符

> 当转换后的sql需要必须移除掉某些？占位符，必须使用下面方法记录
>
> 一般只会在分页语法转换时使用，可以参考com.sangsang.visitor.transformation.oracle2mysql.O2MTfExpressionVisitor

```
com.sangsang.domain.context.TfParameterMappingHolder#recordRemoveParameterMapping
```

### 扩展其他数据库转换

> 框架提供了扩展接口，支持实现其他异构数据库的语法转换

#### 实现步骤

```
步骤1: 创建新的转换模式包
  包路径：com.sangsang.transformation.{模式名}
  示例：oracle2mysql, postgres2mysql 等

步骤2: 配置转换模式
  field.transformation.patternType={模式名}

步骤3: 实现对应的转换器
  参考 mysql2dm 包下的实现，在步骤1创建的包路径下，实现自己的语法转换逻辑
  实现列、表、函数、表达式等转换器
```

#### 配置示例

```properties
# Oracle 转 MySQL
field.transformation.patternType=oracle2mysql

# PostgreSQL 转 MySQL
field.transformation.patternType=postgres2mysql
```

## 七、不兼容场景

| 限制项 | 说明 |
|:---|:---|
| JSQLParser 版本 | 基于 jsqlparser4.9 解析，此版本不支持的语法无法兼容 |
| mysql2dm不兼容<br>mysql的 on duplicate key update | 达梦数据库中尚未找到合适的自动转换语法<br>使用MERGE INTO的话，存在多个唯一约束的话，无法自动判断用哪个，且实现复杂，故不兼容 |

## 八、常见问题

### Q1: 如何确认哪些 SQL 被转换了？

> 框架提供了debug日志，可开启 DEBUG 日志查看转换详情

```properties
# 开启 SQL 转换日志
logging.level.com.sangsang=DEBUG
```

### Q2: 部分 SQL 没有自动转换怎么办？

#### 情况一：语法未覆盖

> 框架未覆盖此语法转换

**解决方案**：参考"进阶使用 → 自定义语法转换器"自行实现
