

# 字段加解密使用文档

>  零侵入式敏感数据加密解决方案

---

## 一、使用场景

> 等保、数据合规等敏感数据加密存储需求

```
┌─────────────────────────────────────────────────────────────┐
│                                                          │
│  ✅  项目快速加密改造                                    │
│  ✅  不改动任何业务代码                                   │
│     • controller 层无需改动                                │
│     • service 层无需改动                                  │
│     • mapper 层无需改动                                  │
│     • 入参响应无需改动                                    │
│                                                          │
└─────────────────────────────────────────────────────────────┘
```

## 三、模式介绍

> 本框架提供两种加密模式，根据项目场景灵活选择
>
> 框架主打的是db模式，推荐使用db模式

### db 模式


![推荐](https://img.shields.io/badge/推荐-高-brightgreen)

#### 工作原理

> 使用数据库本身的加解密函数，自动改写执行 SQL

```sql
-- 原始 SQL
SELECT user_name, phone FROM tb_user WHERE phone = ?

-- 框架自动转换后
SELECT
    数据库解密函数(phone) AS phone,
    user_name
FROM tb_user
WHERE phone = 数据库加密函数(phone)
```

#### 优缺点对比

| 优点 ✅ | 缺点 ❌ |
|:---|:---|
| 加解密场景适应性强 | 原生函数支持有限 |
| **支持密文模糊查询** | 增加数据库计算开销 |
| 支持列运算 | - |
| 绝大多数语法兼容 | - |

> **提示**：可通过 UDF 扩展加密算法，群文件有 MySQL SM4 扩展包

---

### pojo 模式

#### 工作原理

> 不改动执行的 SQL，自动对 SQL 执行的入参和响应进行加解密处理

```
┌─────────────────────────────────────────────────────────────┐
│                                                          │
│  原始 SQL 保持不变                                       │
│         ↓                                                │
│  SQL 语法解析识别加密字段                                 │
│         ↓                                                │
│  Java 层自动加解密入参和响应                              │
│                                                          │
└─────────────────────────────────────────────────────────────┘
```

#### 优缺点对比

| 优点 ✅ | 缺点 ❌ |
|:---|:---|
| Java 算法选择性强 | 部分列运算场景不兼容 |
| 无数据库计算开销 | 常规算法无法密文模糊查询 |
| 配置简单 | - |

## 四、快速接入

### Step 1: 引入依赖

```xml
<!-- 核心依赖 -->
<dependency>
    <groupId>io.gitee.tired-of-the-water</groupId>
    <artifactId>sangsang-core</artifactId>
    <version>${version}</version>
</dependency>

<!-- 注解模块（可选，实体类模块无mybatis依赖时对实体类模块使用）引入了encryptor-core就不需要了 -->
<dependency>
    <groupId>io.gitee.tired-of-the-water</groupId>
    <artifactId>sangsang-annos</artifactId>
    <version>${version}</version>
</dependency>
```

### Step 2: 开启配置

> 💡 **提示**
>
> • 框架会从 scanEntityPackage 扫描的路径中缓存需要密文存储的字段
>
> • DB 模式默认使用 MySQL 库函数，其他函数见"进阶使用"

```properties
# 配置 @TableName 标注的实体类扫描路径
sangsang.scanEntityPackage[0]=com.sangsang.*.entity

# 加密模式选择：db 或 pojo
sangsang.encryptor.patternType=db

# 自定义秘钥（可选，有默认值）
sangsang.encryptor.secretKey=TIREDTHEWATER
```

### Step 3: 字段标注


> ⚠️ **注意**
>
> • 若项目无实体类，在扫描路径下创建，仅保留需标注的字段
>
> • 记得使用 @TableName 标注类表明是哪张表

```java
@Data
@TableName(value = "tb_user")
public class UserEntity extends BaseEntity {

    @TableField(value = "phone")
    @FieldEncryptor  // 标识字段需要加密存储
    private String phone;
}
```

### Step 4: 验证


```bash
# 全局 MyBatis 执行的 SQL
# 涉及到 tb_user 的 phone 字段
# 均可实现加密存储、解密查询 ✅
```

## 五、不兼容场景

### 通用限制（DB & Pojo 模式）

| 限制项 | 说明                                           |
|:---|:---------------------------------------------|
| JSQLParser 语法限制 | 基于 jsqlparser4.9 解析，此版本不支持的语法均无法兼容           |
| MyBatis-Plus的saveBatch() 方法 | 遇到此类场景手动改写法                                  |
| INSERT into 表 VALUES (xxx,xxx) | 表后未跟具体字段<br>在默认开启sangsang.autoFill=true时可以支持 |

### DB 模式限制

```sql
-- 不支持以下语法
INSERT INTO 表1(xxx, xxx)
SELECT * FROM 表2

-- 限制说明：表1使用框架功能，表2未使用时无法处理
-- 原因：未使用的表不缓存字段信息，减少内存占用，所以未做顺序对应
```

### Pojo 模式限制

| 限制项 | 示例 |
|:---|:---|
| ${} 值处理 | `${param}` 无法加解密 |
| 列运算结果 | `SELECT concat(phone,"---") AS ph` 无法解密 ph |
| 同参数多字段 | `WHERE phone = #{p} AND name = #{p}` name和phone加密算法不一致 |
| 字段直接对应 | `INSERT INTO ... SELECT ...` 不经 Java 层 |

### 自定义加密算法

> 当 Spring 容器中只有一种算法时，`@FieldEncryptor` 无需手动指定

#### DB 模式自定义

> **⚠️ 注意**
>
> • 引用 jsqlparser 包：`com.shade.net.sf.jsqlparser`
>
> • 参考实现：`com.sangsang.encryptor.db.DefaultDBFieldEncryptorPattern`
>
> • 有特殊场景可以群里讨论

```java
@Component
@DefaultStrategy  // 多算法时标注默认算法
public class CustomDBEncryptorStrategy implements FieldEncryptorStrategy<Expression> {

    @Autowired
    private SangSangProperties sangSangProperties;

    @Override
    public Expression encryption(Expression oldExpression) {
        // TODO 加密逻辑，建议使用 sangSangProperties 配置的秘钥
        return null;
    }

    @Override
    public Expression decryption(Expression oldExpression) {
        // TODO 解密逻辑
        return null;
    }
}
```



#### Pojo 模式自定义

```java
@Component
@DefaultStrategy  // 多算法时标注默认算法
public class CustomPojoEncryptorStrategy implements FieldEncryptorStrategy<String> {

    @Autowired
    private SangSangProperties sangSangProperties;

    @Override
    public String encryption(String s) {
        // TODO 加密逻辑
        return null;
    }

    @Override
    public String decryption(String s) {
        // TODO 解密逻辑
        return null;
    }
}
```

### 分库分表集成

> 避免为每张分表创建实体类和重复标注
>
> ```
> 栗子："stat_vehicle_alarm"这张表是分表前的表名。
>  - 实际数据库中不存在这张表
>  - 数据库表名是"stat_vehicle_alarm_0101","stat_vehicle_alarm_0102"等根据天进行了分表的
>  - 实现ShardingTableStrategy接口，在里面配置根据原表名获取到分表的表名的规则
> ```

```java
// 原表实体类标注
@ShardingTableEncryptor(value = ShardingTableByDay.class)
@TableName("stat_vehicle_alarm")
public class StatVehicleAlarmEntity {
    // 字段定义
}

/* 分表规则实现
* 栗如：指定根据
*/
public class ShardingTableByDay implements ShardingTableStrategy {

    private static final LocalDate BASE_DATE = LocalDate.parse("2000-01-01");
    private static final DateTimeFormatter DATE_FMT = DatePattern.createFormatter("MMdd");
    private static final List<String> SUFFIX;

    static {
        SUFFIX = new ArrayList<>();
        for (int i = 0; i < 366; i++) {
            SUFFIX.add(LocalDateTimeUtil.format(BASE_DATE.plusDays(i), DATE_FMT));
        }
    }

    @Override
    public List<String> getShardingTableName(String prefix) {
        return SUFFIX.stream()
            .map(suffix -> prefix + suffix)
            .collect(Collectors.toList());
    }
}
```

## 七、常见问题

### Q1: 项目未使用 MyBatis-Plus，如何接入？

#### 情况一：有权限读取表结构

```properties
# 保持默认配置
sangsang.autoFill=true
```

> 在扫描路径下创建 `@TableName` 实体类，仅写入需要密文存储的字段进行标注

#### 情况二：无权限读取表结构

**方案一：使用代码生成器**

> 略


**方案二：使用本项目工具**

```java
EntityGenerateUtil.generateEntity(GenerateDto.builder()
    .packageName("com.sangsang.es.entity")      // 生成实体类包名
    .outputDir("D:\\resource\\draft\\test")      // 生成路径
    .url("jdbc:mysql://127.0.0.1:3306/db")       // 数据库地址
    .username("username")                         // 用户名
    .password("password")                         // 密码
    .catalog("sjj-d")                            // 目录（可选）
    .schemaPattern(null)                          // Oracle传用户名，MySQL传null
    .tableNamePattern("%")                         // %匹配任意字符
    .build());
```



---

### Q2: 实体类字段不写全有什么影响？

| 模式 | 影响 |
|:---:|:---|
| **DB 模式** | ⚠️ 必须写全所有字段，`SELECT *` 场景会导致字段缺失 |
| **Pojo 模式** | ✅ 仅需标注使用的字段 |

