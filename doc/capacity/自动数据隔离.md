

# 自动数据隔离

### 零侵入式数据权限隔离解决方案

---

## 一、使用场景

> 多租户、数据权限等业务数据隔离需求
>
> - 在一个系统中，同一个业务模块（一张表）对应的数据权限规则应该是固定的
> - 常规方案中需要按照功能标注的方向有些走偏了
> - 数据权限就应该一处标注，全局生效

### 核心价值

```
┌─────────────────────────────────────────────────────────────┐
│                                                          │
│  ✅  无需手动拼接 SQL 条件                              │
│  ✅  全局自动拼接隔离条件                                 │
│     • controller 层无需改动                             │
│     • service 层无需改动                                │
│     • mapper 层无需改动                                │
│     • 一处标注，全项目生效                              │
│                                                          │
└─────────────────────────────────────────────────────────────┘
```

## 二、方案对比

| 对比项 | 自定义注解方案 | MyBatis-Plus | **field-encryptor** |
|:---:|:---:|:---:|:---:|
| **使用方式** | 方法/Mapper 注解标注 | DataPermissionHandler | **实体类字段标注** |
| **代码侵入** | 高 | 中 | **无** |
| **改造效率** | 每个接口需标注 | 需封装处理,并了解sql解析语法 | **简单标注** |
| **项目规范** | 不统一，容易遗漏 | 需额外开发 | **统一规范** |
| **扩展性** | 弱 | 中 | **强** |
| **兼容性** | 高，所有语法都能改<br>做差异化处理 | 中，部分冷门语法不支持 | 高，正常语法都兼容 |

## 三、工作原理

> 框架基于 MyBatis 拦截器 + JSQLParser 解析 SQL，自动拼接数据隔离条件

```
┌─────────────────────────────────────────────────────────────┐
│                                                          │
│  业务代码执行 SQL                                         │
│         ↓                                                │
│  MyBatis 拦截器拦截                                   │
│         ↓                                                │
│  JSQLParser 解析 SQL 语法树                            │
│         ↓                                                │
│  识别表的数据隔离策略                                     │
│         ↓                                                │
│  自动拼接隔离条件到 WHERE 子句                             │
│         ↓                                                │
│  输出拼接后的 SQL 执行                                   │
│                                                          │
└─────────────────────────────────────────────────────────────┘
```

### 效果示例

> 场景：`tb_user` 表按 `org_id` 进行数据隔离

**原始 SQL**
```sql
SELECT * FROM tb_user WHERE phone = 'xxx' OR name = 'yyy'
```

**自动拼接隔离条件后**
```sql
-- PC 端
SELECT * FROM tb_user
WHERE (phone = 'xxx' OR name = 'yyy')
  AND org_id = 登录用户组织id

-- APP 端
SELECT * FROM tb_user
WHERE (phone = 'xxx' OR name = 'yyy')
  AND id = 登录用户id
```

## 四、支持的隔离模式

| 隔离模式 | 说明 | 示例 |
|:---:|:---|:---|
| **=** | 精确匹配 | `org_id = 123` |
| **IN** | 多值匹配 | `org_id IN (1, 2, 3)` |
| **LIKE** | 模糊前缀匹配 | `org_seq LIKE '123%'` |

> **提示**：支持根据业务场景灵活切换隔离模式

## 五、快速接入

### Step 1: 引入依赖

```xml
<!-- 核心依赖 -->
<dependency>
    <groupId>io.gitee.tired-of-the-water</groupId>
    <artifactId>encryptor-core</artifactId>
    <version>${version}</version>
</dependency>
```

### Step 2: 开启配置

> 💡 **提示**
>
> • 框架会从 scanEntityPackage 扫描的路径中缓存表字段信息
>
> • 默认仅对 SELECT 语句生效，对 DML 生效需额外配置

```properties
# 配置 @TableName 标注的实体类扫描路径
field.scanEntityPackage[0]=com.sangsang.*.entity

# 开启数据隔离
field.isolation.enable=true
```

### Step 3: 自定义数据隔离策略

> **⚠️ 注意**
>
> • Spring 容器中只有一个策略时，自动作为默认策略
>
> • 多个策略时，使用 `@DefaultStrategy` 标注默认策略
>
> • 通过ThreadLocal获取当前用户时，若有父子线程+线程池的场景，使用阿里的TransmittableThreadLocal

```java
package xxx.strategy;

import com.sangsang.domain.enums.IsolationRelationEnum;
import com.sangsang.domain.strategy.isolation.DataIsolationStrategy;
import org.springframework.stereotype.Component;

/**
 * 数据隔离策略示例
 */
@Component
@DefaultStrategy  // 多策略时标注默认策略
public class OrgIsolationStrategy implements DataIsolationStrategy<Long> {

    @Override
    public String getIsolationField(String tableName) {
        // 返回需要进行数据隔离的字段名
        // 一般从 ThreadLocal 中获取当前登录用户信息
        // 根据不同用户类型返回不同隔离字段
        return "org_id";
    }

    @Override
    public IsolationRelationEnum getIsolationRelation(String tableName) {
        // 支持 "="、"in"、"like 'xxx%'" 三种模式
        return IsolationRelationEnum.EQUALS;
    }

    @Override
    public Long getIsolationData(String tableName) {
        // 从 ThreadLocal 中获取当前登录用户的隔离字段值
        // 如：当前登录用户的组织 ID
        return 777777777L;
    }
}
```

### Step 4: 实体类标注

> 标注在实体类上，一个实体类表示一种抽象数据

```java
@Data
@TableName("tb_order")
// 使用默认策略进行数据隔离
@DataIsolation
public class OrderEntity extends BaseEntity {

    @TableField("org_id")
    // 实体类中必须包含隔离字段
    private Long orgId;
}
```

### Step 5: 验证

```bash
# 所有涉及 tb_order 的查询 SQL
# 自动拼接数据隔离条件 ✅
```

### 禁用数据隔离

> 当某些场景不需要数据隔离时

#### 单个 SQL 禁用

```java
@ForbidIsolation
List<OrderEntity> getPage();
```

#### Service 方法禁用

```java
@ForbidIsolation
public void test() {
    // 各种 SQL 都不需要数据隔离
}
```

---

### 配置多策略关系

#### 多表隔离条件关系

> 一个 SQL 中多个表均有数据隔离条件，在全局配置中设置

```properties
# 不同表的隔离条件之间的关系，默认是 and
field.isolation.conditionalRelation=or
```

#### 单表多策略关系

```java
@TableName("sys_user")
@DataIsolation(
    conditionalRelation = IsolationConditionalRelationEnum.OR,
    value = {策略1.class, 策略2.class}
)
public class SysUserEntity {
    // 字段定义
}
```

---

### IN 分段处理

> Oracle 默认超过 1000 条会报错

```properties
# 分段大小，默认 500
field.isolation.inRelationSubsectionSize=500
```

**分段前**
```sql
id IN (xxx, xxx, ..., 共计1000个)
```

**分段后**
```sql
(id IN (xxx, xxx, ..., 共计500个)
 OR id IN (xxx, xxx, ..., 共计500个))
```

### 复杂场景示例

#### 场景描述

> PC端：
>
> - 组织架构分为 一级组织 → 二级组织 → 三级组织
>
> - 当前组织可以看当前组织和所有下级组织的数据
>
> C端：
>
> - C端用户只能看到自己的（比如司机只能看到自己的订单）
>
> 表结构设计：
>
> 我们创建一个字段叫org_seq ，存储格式是
>
> - 三级组织： 一级组织id-二级组织id-当前组织id
> - 二级组织：一级组织id-当前组织id
> - 一级组织：当前组织id
>
> 每个单据上会有一个司机id
>
> - driver_id

| 客户端 | 组织架构 | 隔离方式 |
|:---:|:---|:---|
| **PC 端** | 一级组织 → 二级组织 → 三级组织 | `org_seq LIKE '当前组织前缀%'` |
| **APP 端** | 司机独立 | `driver_id = 当前司机ID` |

#### 策略实现

```java
@Component
public class MultiClientIsolationStrategy implements DataIsolationStrategy<String> {

    @Override
    public String getIsolationField(String tableName) {
        // 从 ThreadLocal 获取用户类型
        UserType userType = UserContext.getUserType();

        if (userType == UserType.DRIVER) {
            return "driver_id";
        } else {
            return "org_seq";
        }
    }

    @Override
    public IsolationRelationEnum getIsolationRelation(String tableName) {
        UserType userType = UserContext.getUserType();

        if (userType == UserType.DRIVER) {
            return IsolationRelationEnum.EQUALS;
        } else {
            return IsolationRelationEnum.LIKE;
        }
    }

    @Override
    public String getIsolationData(String tableName) {
        UserType userType = UserContext.getUserType();

        if (userType == UserType.DRIVER) {
            return UserContext.getDriverId();
        } else {
            return UserContext.getOrgSeq();
        }
    }
}
```

## 八、常见问题

### Q1: 如何实现不同用户类型使用不同隔离字段？

> 参考"进阶使用 → 复杂场景示例"

从 `ThreadLocal` 中获取当前用户类型，在策略中返回不同的隔离字段和关系模式。

---

### Q2: 异步场景下隔离条件丢失怎么办？



> **⚠️ 注意**
>
> • 数据隔离信息一般存于 `ThreadLocal`
>
> • 异步场景存在父子线程值传递问题
>
> • 建议使用 `TransmittableThreadLocal` 替换 `ThreadLocal`
>
> • InheritableThreadLocal 无法解决线程池场景下的父子线程传值问题
